## Domain Ownership Mapping

- `runtime_platform`
  - Metrics plumbing and service-level runtime wiring
  - Distributed runtime topology/bootstrap hooks
- `storage_engine`
  - Lifecycle rule execution and retention behavior
  - Replication and distributed storage primitives
  - Erasure-coding correctness/performance hardening
- `cluster_transport_security`
  - Peer authentication and node-to-node trust boundaries
  - Trusted internal forwarding/replication transport contracts
- `cluster_metadata_plane`
  - Cluster-authoritative bucket metadata and listing correctness model
  - Metadata convergence and token-stable distributed listing behavior
- `s3_auth_sigv4`
  - Multi-user auth model foundation for request signing/verification
- `s3_api_surface`
  - CORS behavior and S3-facing lifecycle APIs
  - Distributed/replication-aware API behavior
- `console_api`
  - Multi-user session/auth flows
  - JSON endpoints for lifecycle/metrics/admin workflows
- `web_console_ui`
  - Multi-user and policy-oriented UX
  - Lifecycle/metrics UI workflows
- `quality_harness`
  - End-to-end regression and compatibility tests for each roadmap item

## Delivery Order (Roadmap-Aware)

1. CORS (low-risk, high-utility compatibility gap)
2. Versioning hardening completion and test expansion
3. Lifecycle rules (storage + API + console/UI)
4. Multi-user support (auth + console + UI + config)
5. Metrics (runtime + API exposure + UI)
6. Replication and distributed mode (largest cross-domain scope)

## Explicit Near-Term Gaps (March 2, 2026)

- Storage decomposition (`storage_engine`):
  - `src/storage/filesystem.rs` staged split Stages 1-4 are complete and currently sufficient; Stage 5 facade-thinning is deferred unless measurable coupling/regression pressure appears.
- Console contract typing (`console_api`):
  - Console handler responses are now migrated to typed DTO contracts across system, buckets, versions, objects, lifecycle, presign, and auth/session paths; keep new endpoint work on typed DTOs and contract tests to prevent regression into ad-hoc payload maps.
- Membership protocol semantics (`runtime_platform`):
  - Runtime now has a typed `MembershipEngine` boundary with status surfaced in topology/health/metrics (`engine`, `ready`, `converged`, `lastUpdateUnixMs`).
  - Runtime health/metrics now also surface explicit convergence quality reason contracts (`membershipConvergenceReason`, `maxio_membership_convergence_reason_info{reason=\"...\"}`) so operators can distinguish engine-placeholder vs connectivity vs view-mismatch non-convergence.
  - Console system `metrics` and `summary.metrics` now mirror `membershipConvergenceReason`, keeping admin API convergence diagnostics aligned with runtime health/metrics contracts.
  - `gossip` now runs through an active experimental engine (`gossip-experimental`, ready) with probe-driven convergence/discovery wiring; `raft` remains the unimplemented placeholder engine (degraded).
  - gossip discovery now also performs fail-safe stale-peer pruning when direct probe failures are corroborated by successful same-view peer snapshots (failed peer absent from responder-advertised topology), reducing stale membership drift in long-running clusters.
  - Runtime now exposes authenticated join-apply control-plane wiring (`POST /internal/cluster/join`) that reuses join authorization and applies peer-add transitions with optional stale-state preconditions (`expectedMembershipViewId`, `expectedPlacementEpoch`).
  - Internal membership update control-plane now rejects stale topology updates via optional preconditions (`expectedMembershipViewId`, `expectedPlacementEpoch`) with deterministic `409 precondition_failed`, reducing stale-writer race risk during live control-plane operations.
  - Runtime membership apply transitions are now serialized in-process (async mutex guard in `apply_membership_peers`) so concurrent join/discovery/update calls preserve monotonic placement-epoch progression and deterministic persisted state ordering.
  - Runtime now also runs a background static-bootstrap membership convergence probe worker; observed convergence state is persisted in runtime memory and surfaced via topology/health/metrics without requiring endpoint polling side effects.
  - Static-bootstrap convergence workers now also ingest peer-advertised topology hints from peer `/healthz` payloads (`clusterId`, `clusterPeers`) and can apply additive discovered peer-set updates autonomously through runtime membership apply paths.
  - Discovery apply now fail-closes on peer-reported cluster-id mismatch and invalid discovered peer normalization, preventing malformed snapshot drift from mutating local topology.
  - Discovery apply now also fails closed when peer snapshots omit `clusterId` or `membershipViewId`, so ambiguous `/healthz` payloads cannot influence live discovered peer-set expansion.
  - Membership-update peer fanout failures are now persisted to `.maxio-runtime/pending-membership-propagation-queue.json` and replayed by a background worker (`spawn_pending_membership_propagation_replay_worker`) so control-plane convergence survives process restarts and transient peer outages.
  - Membership-update fanout targeting now uses the canonical union of previous + updated peer sets (deduped, local-node excluded), so remove-peer and transition-to-standalone updates still notify recently removed peers instead of fanouting only to the post-update peer set.
  - Runtime health/metrics now expose typed pending membership-propagation queue and replay-worker observability (`checks.pendingMembershipPropagationQueueReadable`, `pendingMembershipPropagationBacklog*`, `pendingMembershipPropagationReplay*`, `maxio_pending_membership_propagation_*`) so control-plane backlog/replay convergence state is operator-visible.
  - Membership-propagation replay now runs through bounded prioritized worksets (`PENDING_MEMBERSHIP_PROPAGATION_REPLAY_BATCH_SIZE`), with deferred due-operation telemetry (`maxio_pending_membership_propagation_replay_deferred_operations_total`) to keep replay cycles bounded under sustained backlog pressure.
- Cluster trust boundary (`cluster_transport_security`) [active]:
  - internal forwarding now supports shared-token trust gating via `MAXIO_CLUSTER_AUTH_TOKEN` and strips spoofable inbound auth token headers before forwarding.
  - internal-operation headers are only trusted when forwarded marker + configured token validation passes.
  - S3 forwarding/internal-replica trust now also requires direct sender allowlist membership in shared-token mode (non-cluster senders with matching token are treated as untrusted).
  - shared-token sender/local/allowlist binding now canonicalizes peer identities case-insensitively, so case-variant node IDs do not bypass self-sender checks or distort allowlist binding/readiness status.
  - forwarded-by chains now reject duplicate peer hops (in addition to malformed and over-hop-limit chains) to tighten loop/replay containment.
  - runtime health/metrics now expose trust posture contracts (`clusterAuthTrustModel`, `clusterAuthTransportIdentity`, `clusterPeerAuthIdentityBound`, `maxio_cluster_peer_auth_trust_model_info`, `maxio_cluster_peer_auth_identity_bound`) for operator visibility.
  - runtime distributed readiness now also requires sender-allowlist trust binding (`checks.clusterPeerAuthSenderAllowlistBound`), and metrics now surface `maxio_cluster_peer_auth_sender_allowlist_bound` to prevent false healthy status in compatibility/no-allowlist modes.
  - runtime shared-token diagnostics now distinguish explicit binding faults (for example invalid local node-id shape) from generic no-allowlist posture, reducing ambiguous transport-trust warnings.
  - distributed internal control-plane hard-gating now fails closed when shared-token peer auth is not configured:
    - `/internal/cluster/join/authorize` and `/internal/cluster/membership/update` now return deterministic `503` with `reason=cluster_auth_token_not_configured`.
    - integration regressions now lock both paths under runtime and cluster transport-security domain runners.
  - live membership-update control-plane now also enforces forwarded sender allowlist trust in shared-token mode:
    - `/internal/cluster/membership/update` now requires trusted `x-maxio-forwarded-by` sender identity in addition to join-header authorization.
    - unknown/non-allowlisted forwarded senders now fail closed with deterministic `403 unauthorized` + `authReason=sender_not_in_allowlist`.
  - live join-apply control-plane now enforces the same forwarded sender allowlist trust in shared-token mode:
    - `/internal/cluster/join` now requires trusted `x-maxio-forwarded-by` sender identity in addition to join-header authorization.
    - unknown/non-allowlisted forwarded senders now fail closed with deterministic `403 unauthorized` + `authReason=sender_not_in_allowlist`.
  - join-authorization nonce policy now enforces a minimum nonce length floor (8 chars) and rejects short nonces as `invalid_join_nonce`.
  - cluster transport auth now also exposes explicit legacy forwarding-header detection (`contains_legacy_internal_forwarding_headers`) while keeping active SigV4 sanitization gates trust-marker scoped, preserving distributed forwarding compatibility for externally signed requests.
  - mTLS readiness now supports optional local certificate SHA-256 pin validation (`MAXIO_CLUSTER_PEER_TLS_CERT_SHA256`) with explicit fail-closed mismatch/invalid-pin diagnostics (`certificate_fingerprint_pin_mismatch`, `certificate_fingerprint_pin_invalid`) surfaced in health/metrics; shared-token transport consumers now also enforce node-id binding pin posture (`node_identity_binding_pin_required`) before enabling mTLS forwarding/fanout transport.
  - mTLS readiness now also fail-closes expired/not-yet-valid local certs (`certificate_validity_window_invalid`), and handshake attestation now fail-closes expired/not-yet-valid peer certs (`peer_certificate_validity_window_invalid`) before accepting node-identity matches.
  - Runtime peer transport now also loads configured client certificate/private-key identity into HTTPS peer calls (`reqwest::Identity::from_pkcs8_pem`) for control-plane probe/fanout/replay paths, and transport-identity node binding now verifies expected node host against pinned certificate SAN/CN with fail-closed mismatch semantics (`node_identity_certificate_mismatch`).
  - handshake-time remote peer certificate verification and per-peer certificate-to-node-id attestation are now active on runtime peer control-plane/probe paths plus S3 and console internal forwarding/fan-in transport paths.
  - next step is hardening certificate lifecycle policy (pin rotation/rollover guidance, revocation handling, and strict mTLS-only production posture).
  - runtime peer control-plane/probe calls now consume shared transport-scheme wiring (instead of hardcoded `http://`) so static-bootstrap peer probes (`/healthz`), membership-update propagation fanout (`/internal/cluster/membership/update`), and rebalance receive pulls switch to `https` when mTLS path-mode transport is configured/ready and fail-close client initialization when that transport posture is misconfigured.

## Distributed Implementation Exit Targets (Execution-Aligned)

To avoid "hardening-only loops", domain cycles should primarily burn down these execution targets:

1. **Live membership engine target (`runtime_platform` + `cluster_transport_security`)**
   - Deliverable: runtime-updatable membership view/epoch transitions (not restart-derived only). Baseline control-plane apply path is now active via authenticated internal join/apply and membership-update endpoints; static-bootstrap additive discovery from peer health snapshots is now active; autonomous non-static engine convergence remains.
   - Exit criteria: membership updates change routing decisions live, epoch/view mismatches fail closed, and join/auth flows are bound to engine state.
2. **Metadata-authoritative listing target (`cluster_metadata_plane` + `s3_api_surface` + `console_api`)**
   - Deliverable: one active cluster-authoritative metadata strategy wired into runtime/API list paths.
   - Exit criteria: distributed list/version responses are complete for the chosen strategy (or fail closed when strategy is unready), with stable token continuity under snapshot/view changes.
3. **Durability completion target (`storage_engine` + `s3_api_surface`)**
   - Deliverable: degraded-success durability path backed by durable pending-replication execution.
   - Exit criteria: pending fanout is persisted, replayed, observable, and converges replicas after transient peer failure/restart scenarios.
4. **Rebalance execution target (`storage_engine` + `runtime_platform` + `console_api`)**
   - Deliverable: runtime executor for transfer plans produced by typed rebalance planning (object-scope execution now active).
   - Exit criteria: join/leave transitions migrate ownership data with idempotent progress tracking and operator-visible state, including chunk-scope parity and replay-cycle observability/completion semantics.

Execution guardrail:
- Hardening-only cycles are allowed when they close a security/correctness blocker for one of the four targets above; otherwise, prioritize direct implementation progress on the next unsatisfied target.
